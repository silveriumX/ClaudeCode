## 1. БИЗНЕС-КОНТЕКСТ

### Что за бизнес
- ~7 магазинов на маркетплейсах Wildberries + Ozon
- Разные ИП/юрлица, часть с НДС, часть без
- Есть партнёры-владельцы отдельных магазинов и свои магазины
- Команда распределённая: РФ, BY, KZ
- Зарплаты выплачиваются в рублях, BYN, KZT, USDT

### Текущий учёт
- Основной инструмент — Google Sheets
- Каждый магазин ведёт свой журнал операций (горизонтальная структура: банки в столбцах)
- Telegram-бот (xRocket) принимает заявки на оплату и пишет в таблицу
- Финотчёты WB/Ozon выгружаются вручную
- USDT-журнал ведётся вручную, общий на всю компанию
- Сверок нет

### Главные проблемы
1. **USDT-потоки непрозрачны** — ~10 кошельков (без биржевых), только собственник имеет доступ, ретроспективу сложно восстановить
2. **Нет контроля за движением по счетам** — бизнес-банк + 5-10 физ.карт на магазин, не видно без выписок
3. **Нет сверок** — финотчёт маркетплейса ↔ поступление на р/с ↔ фактические расходы
4. **Данные заполняются разными людьми** — нет гарантии полноты и корректности
5. **Невозможно посчитать реальный PnL** по магазину и по компании

## 2. СПРОЕКТИРОВАННАЯ АРХИТЕКТУРА

### 2.1 Принципы
- Каждая операция записывается один раз в одном месте (без дублей)
- 4 уровня: Источники данных → Журналы операций → Сверки и контроль → PnL + Дашборд
- Google Sheets — основной инструмент, с возможностью миграции позже
- Доступ разграничен по ролям

### 2.2 Структура Google Sheets

**Файл на каждый магазин: «Магазин [Название]»**

| Лист | Содержание | Кто заполняет |
|------|-----------|---------------|
| Блок Маркетплейс | Финотчёт WB/Ozon: выручка, комиссии, логистика, штрафы, хранение, реклама | Менеджер магазина |
| Бизнес-банк | Поступления на р/с, комиссии, оплаты: фулфилмент, закупки, декларации, аренда, зарплаты, налоги | Менеджер магазина |
| Наличные | Снятие с карт → приход. Расход: доставки, товары, зарплаты наличными | Ответственный за кассу |
| Физ.банки | Движение по физ.картам: переводы, оплаты, снятия, подписки | Менеджер / владелец |
| Зарплаты | Сотрудники магазина: фикс, бонусы, %, валюта, даты | Менеджер |
| Себестоимость | Товарные позиции, партии, цены закупки, остатки | Менеджер |
| Налоги | УСН, НДС, страховые, НДФЛ — ставки и расчёт | Бухгалтер |
| Сверка | Автоматический лист — расхождения подсвечиваются | Автоматика |

**Общие файлы:**
- **USDT-журнал** — все крипто-операции всех магазинов
- **Консолидация** — сводный PnL по всем магазинам
- **Дашборд** — ключевые метрики, алерты

### 2.3 Формат записи транзакции (новый, вертикальный)

| Поле | Описание |
|------|----------|
| ID | Автоматический уникальный номер |
| Дата | Дата операции |
| Тип | Доход / Расход / Перевод / Конвертация |
| Категория | Из списка 14 категорий |
| Подкатегория | Уточнение |
| Сумма | Числовое значение |
| Валюта | RUB / USDT / BYN / KZT |
| Счёт-источник | Откуда ушли деньги |
| Счёт-получатель | Куда пришли (для переводов) |
| Контрагент | Кому / от кого |
| Комментарий | Свободный текст |
| Кто внёс | Автоматически |

### 2.4 Потоки денег

**Основной поток (маркетплейс):**
```
Покупатель → WB/Ozon → Финотчёт → Перечисление на р/с → Расходы
```

**Наличные:**
```
Р/с → Бизнес-карта → Снятие → Касса склада → Расходы наличными
```

**USDT:**
```
Р/с → Физ.карта → Снятие нал. → Покупка USDT → USDT-кошелёк → Оплата (зарплата/услуга)
```

### 2.5 Ключевое правило: Перевод vs Расход vs Конвертация

- **Перевод** = перемещение между своими счетами. НЕ попадает в PnL. Примеры: р/с → физ.карта, физ.карта → касса
- **Расход** = деньги ушли третьему лицу (сотрудник, поставщик). Попадает в PnL
- **Конвертация** = подтип перевода с изменением валюты. Фиксируется курс. Не попадает в PnL, но изменяет валютные остатки

**Пример цепочки:**
1. Р/с → физ.карта (Перевод, 100к RUB)
2. Снятие наличных (Перевод, 100к RUB)
3. Покупка USDT (Конвертация, 100к RUB → 1200 USDT, курс 83.3)
4. Оплата дизайнеру (Расход, 500 USDT, категория «Зарплата»)

**В PnL попадает только строка 4.** Остальное — перемещение внутри «кошелька компании».

### 2.6 Категории расходов (14 штук)

1. **Закупка товара** — товар, образцы, доставка от поставщика
2. **Фулфилмент** — упаковка, маркировка, приёмка, хранение (не WB)
3. **Логистика WB/Ozon** — комиссия логистики маркетплейса, штрафы
4. **Комиссия маркетплейса** — комиссия с продаж, реклама, продвижение
5. **Зарплата** — фикс, бонусы, %, НДФЛ, взносы
6. **Аренда** — склад, офис, коммуналка
7. **Налоги и сборы** — УСН, НДС, патент, страховые ИП
8. **Сертификация** — декларации, протоколы, ЭЦП
9. **Маркетинг (внешний)** — дизайн, фото, видео, блогеры, не маркетплейс
10. **IT и сервисы** — подписки, разработка, API, CRM
11. **Офис и хозяйство** — канцелярия, оборудование, мебель
12. **Банковские расходы** — тарифы, комиссии, эквайринг
13. **Представительские** — командировки, подарки, обеды
14. **Прочее** — всё что не попало в категории

> ⚠️ Текущие категории в боте неправильные (созданы скриптом), нужно пересоздать с нуля.

### 2.7 Роли и доступ (4 роли)

| Роль | Доступ | Кто |
|------|--------|-----|
| Собственник | Всё: все магазины, крипта, консолидация, дашборд | Ты |
| Менеджер магазина | Только свой магазин: журнал, зарплаты, себестоимость | Руководители магазинов |
| Бухгалтер | Все магазины: налоги, сверки, выписки | Бухгалтер |
| Исполнитель | Ограниченный ввод: бот заявок, касса склада | Сотрудники |

Реализация: разграничение через Google Sheets permissions на уровне файлов.

### 2.8 Система сверок (5 точек)

1. **Маркетплейс ↔ Р/с** — сумма «К перечислению» из финотчёта vs. фактическое поступление на р/с
2. **Журнал расходов ↔ Выписка банка** — записи в журнале vs. данные выписки
3. **Расчётный остаток ↔ Фактический остаток** — по каждому счёту (как в текущем «Балансе»)
4. **USDT-журнал ↔ Остаток кошельков** — расчётный остаток по журналу vs. реальный
5. **Бот заявок ↔ Журнал расходов** — каждая одобренная заявка должна иметь запись в журнале

### 2.9 PnL (отчёт о прибылях и убытках)

**По магазину:**
```
Выручка (из финотчёта WB/Ozon)
— Себестоимость товара
— Комиссия маркетплейса
— Логистика WB/Ozon
= Маржинальная прибыль

— Фулфилмент
— Зарплаты (доля магазина)
— Аренда (доля магазина)
— Маркетинг
— IT/сервисы
— Сертификация
— Налоги
— Банковские
— Офис/хозяйство
— Прочее
= Операционная прибыль

— Выплата владельцу
= Чистая прибыль магазина
```

**Консолидированный:** сумма всех магазинов + общекомпанейские расходы.

### 2.10 Дашборд

Метрики: выручка, маржа, PnL по магазину, USDT-баланс, расхождения сверок, топ-категории расходов.
Алерты: расхождение >5%, необычные операции, просроченные сверки.

### 2.11 Процессы

**Ежедневно:** менеджеры вносят операции, кассиры — наличные
**Еженедельно:** бухгалтер делает сверку банков, собственник проверяет USDT
**Ежемесячно:** загрузка финотчётов WB/Ozon, расчёт PnL, начисление зарплат, расчёт налогов

## 3. ТЕКУЩЕЕ СОСТОЯНИЕ (анализ загруженных файлов)

### 3.1 Журнал операций — МН (эталон)

**Файл:** `Журнал_операций___МН.xlsx`
**Структура:** горизонтальная — дата в столбце A, далее пары столбцов «сумма + комментарий» по каждому банку/счёту.

**Листы:**
- Справочник (997 строк × 5 столбцов) — банки, виды операций, категории
- Журнал_НЕАКТ — неактуальный/архивный
- Журнал для заполнения — текущий рабочий лист
- Журнал — основной с данными
- Баланс — сверка расчётного vs. фактического остатка
- Баланс архив
- СВОДы по месяцам (апрель, май, июнь, июль)

**Банки в журнале (16 шт.):** МодульБанк, Тинькофф, Сбербанк, Райффайзен, Газпром, ВТБ и др.
**Формат:** плюсы = поступления, минусы = списания
**Нет категоризации расходов** — только свободный текст в комментариях

**Наблюдения:**
- Хорошая основа для сверки (лист «Баланс»)
- Горизонтальная структура неудобна для анализа — переходим на вертикальную
- Все другие магазины используют такую же структуру (или не ведут журналов вообще)

### 3.2 Крипто-журнал

**Файл:** `Журнал_операций___Крипто_.xlsx`
**Объём:** ~355 строк операций, ~24 USDT-кошелька
**Покрытие:** общий на всю компанию. НЕ включает биржевые кошельки (Bybit, HTX)
**Нет категоризации** и нет привязки к магазинам

**Важно:** около 3 биржевых кошельков (основных 1-2), их стоит включить в систему, но пока непонятно как — данные по ним в отдельном журнале.

### 3.3 Бот выплат (xRocket)

**Файл:** `Выплаты_xRocket_-_рубли_.xlsx`
**Объём:** ~986 строк, 26 пользователей
**Текущие категории (неправильные, созданы скриптом):** Зарплата, Логистика, Маркетинг, Прочее, Услуги, Офис, Разработка, Операционка, Закупки
**Нет привязки к магазинам**

**Назначение:** обслуживает фактически только часть расходов — в основном зарплатные выплаты. Часть зарплат проходит по другим каналам (безнал, вывод на карту владельца ИП).

**Также через бота фиксируются:**
- Наличные расходы склада (кассиры вносят чеки)
- Выплаты ИП-владельцам (перевод ИП → личная карта)

**Техническое:** бот на VPS, редактировать можно абсолютно всё. Есть HTML-схема архитектуры бота (`1770893596951_ARCHITECTURE_DIAGRAM.html`).

### 3.4 Баланс счетов

**Файл:** `Баланс_счетов___Cozy_Space.xlsx`
**Назначение:** ежедневная фиксация фактических остатков по банкам для конкретного магазина (Cozy Space).
**Формат:** даты в строках, банки в столбцах, ручной ввод остатков.

### 3.5 Схема бота

**Файл:** `1770893596951_ARCHITECTURE_DIAGRAM.html`
**Содержание:** визуальная HTML-схема архитектуры Telegram-бота для выплат.
