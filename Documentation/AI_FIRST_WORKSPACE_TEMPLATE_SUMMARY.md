# AI First Workspace Template — конспект документации

> Автор: Seva Ustinov, на основе реального воркспейса Elly Analytics

---

## 1. Идея и цель

**Два сдвига:**

1. **От ChatGPT в браузере к суперагентам (Cursor)**
   AI видит весь контекст бизнеса, не теряется история, не нужно копировать между вкладками.

2. **От личного использования AI к организационному**
   Вся компания в одном контексте: общие промпты, скрипты, воркфлоу, знания переиспользуются между отделами.

**Результат:** не разрозненный ChatGPT у каждого, а единый «AI-организм» компании с переиспользуемым контекстом и процессами.

---

## 2. Что даёт шаблон

- **Продуктивность:** рутина — у AI, люди — на стратегию; у каждого — свой контекст агента.
- **Единое знание:** контекст по отделам (стратегия, продукт, маркетинг, операции и т.д.), общие воркфлоу и прозрачность.
- **Enterprise:** мультирепо, доступ по репозиториям, нормальные git-процессы; заявлено использование в команде 40+ человек.

**Для кого:** растущие компании (10–100+), несколько отделов, руководство, миграция с Notion/Confluence/Google Docs.

## 3. Архитектура: мультирепозитории

```
SharedWorkspace/
├── .cursorrules              # Контекст и переключение отделов
├── README.md
├── setup/                    # Скрипты: клонирование и ежедневный sync
│   ├── clone-all-repos.sh / .ps1
│   └── update-all.sh / .ps1
├── Docs/                     # Отдельные git-репо по отделам
│   ├── Strategy/             # Стратегия, конкуренты, решения
│   ├── Product/              # Роадмап, спецификации, исследования
│   ├── SalesAndMarketing/    # Кампании, контент, продажи
│   ├── Operations/           # Процессы, метрики
│   │   └── Hiring/           # Найм и рекрутинг
│   ├── Finance/              # Модели, прогнозы
│   └── Legal-HR/             # Договоры, HR
├── Dev/                      # Код продукта(ов)
│   └── [ProductName]/
├── Projects/                 # Клиентские проекты (отдельный репо)
└── Presales/                 # Пресейл и предложения
```

**Принципы:**

- **Автономия отделов** — каждый отдел владеет своим репо и воркфлоу.
- **AI context switching** — по запросу («Use strategy context») AI читает `Docs/[Department]/.cursorrules` и ведёт себя как эксперт отдела.
- **Видимость между отделами** — прозрачность при сохранении специализации.
- **Доступ** — через права GitHub (org/repo).
- **Инструменты** — те же, что у разработчиков (Cursor, git).

## 4. Система переключения контекста (Context Switching)

В корневом `.cursorrules` задаётся только **управление воркспейсом и переключение отделов**. Детали по отделу лежат в `Docs/[Department]/.cursorrules`.

**Команды пользователя → действие агента:**

| Фраза | Действие |
|-------|----------|
| "Use strategy context" | Читать `Docs/Strategy/.cursorrules` |
| "Use product context" | Читать `Docs/Product/.cursorrules` |
| "Use marketing context" | Читать `Docs/SalesAndMarketing/.cursorrules` |
| "Use operations context" | Читать `Docs/Operations/.cursorrules` |
| "Use hiring context" | Читать `Docs/Operations/Hiring/.cursorrules` |
| "Use finance context" | Читать `Docs/Finance/.cursorrules` |
| "Use general context" | Без отдела, общие правила воркспейса |

**Правило:** при смене контекста AI должен **прочитать** соответствующий `.cursorrules` отдела и держать этот контекст, пока пользователь явно не переключит.

## 5. Реализация (три фазы)

### Фаза 1: Репозитории

- Создать GitHub Organization.
- Создать репо: Strategy, Product, SalesAndMarketing, Operations, Operations-Hiring, Finance, Legal-HR, Dev-[ProductName], Projects, Presales, SharedWorkspace.
- Клонировать SharedWorkspace, в нём запустить `setup/clone-all-repos.sh` или `.\setup\clone-all-repos.ps1` (предварительно подставить свою org и список Dev-репо в скрипте).

### Фаза 2: Контент и кастомизация

- В корне: заменить `[COMPANY NAME]` и плейсхолдеры в `.cursorrules`, поправить скрипты в `setup/`, обновить README.
- Важно: `cp .gitignore.template .gitignore` — чтобы Docs/Dev/ и т.д. были исключены в основном воркспейсе (они отдельные репо).
- Заполнить каждый отдел контентом (шаблон даёт примеры от Elly Analytics).
- Онбординг команды и права доступа по репо.

### Фаза 3: Инструменты и миграция

- Cursor у всех, импорт workspace.
- Обучение: переключение контекста, ежедневный `update-all`, коммиты по отделам.
- Постепенно: 1–2 недели — пилот (например, Strategy), 3–4 — перенос контента, 5–6 — обучение команды, далее масштабирование.

**Рекомендация из шаблона:** не делать всё вручную — открыть папку в Cursor и попросить агента: «Help me implement the AI First Workspace Template for my company…» и дать ссылку на репо; агент помогает со структурой, скриптами и кастомизацией.

## 6. Скрипты setup

- **clone-all-repos** (bash/PS1): создаёт Docs, Dev, клонирует репо организации; в PS1 нужно задать `$ORG` и `$DEV_REPOS`.
- **update-all** (bash/PS1): ежедневный `git pull` по всем клонированным репо (удобно запускать утром).

Платформы: macOS/Linux (bash), Windows (PowerShell).

## 7. Правила из .cursorrules (важное)

**Режимы воркспейса:**

- **Template Mode** — по умолчанию: всё как примеры/обучение, плейсхолдеры.
- **Real Project Mode** — после того как пользователь называет компанию/проект: убираются шаблонные пометки, подставляются реальные данные, структура сохраняется.

**Git (для агента):**

- Не пушить без явного подтверждения пользователя.
- Не коммитить, пока пользователь не скажет, что логический кусок готов.
- Не редактировать корневой `.cursorrules` без явного запроса.
- Предлагать коммит/пуш в подходящие моменты.

**Антигаллюцинации и источники:**

- Один канонический источник на утверждение — помечать `[CANONICAL]`.
- Остальные ссылки — `[REF: path/file.md#section]`.
- Не дублировать формулировки, использовать ссылки; при изменении каноника — обновлять все `[REF:]`.
- Плейсхолдеры: `[PLACEHOLDER: owner]`; даты «Last Updated».

**Числа и диапазоны:**

- Не полагаться на «мысленную» проверку диапазонов; использовать скрипт `scripts/check_range.py` (например, для бюджетов, метрик здоровья и т.п.).

**Язык:**

- По умолчанию — английский для общефирменных документов; существующие документы можно оставлять на родном языке; следовать явным указаниям пользователя.

## 8. Демо и быстрый старт (DEMO-QUICK-START.md)

Примеры команд для демо:

- Стратегия: «A new competitor appeared, analyze them using our workflow and put it in the appropriate folder».
- Проекты: «Take the latest team call transcript and update project status cards».
- Маркетинг: «Create a summary of which projects are ready for testimonial collection».
- Финансы: «Check if these expense numbers are within budget ranges: …».
- Fireflies: разбор транскрипта звонка (договорённости, ответственные, удовлетворённость клиента).

Демо строится вокруг: обзор структуры → анализ конкурента → обновление статусов по транскрипту → кроссподразделенческий сценарий.

## 9. FAQ (кратко)

- **Не то же ли самое, что Notion + AI?** Нет: здесь AI может писать и выполнять код (скрипты, интеграции, автоматизация), а не только текст и поиск.
- **Надёжно ли выполняет инструкции?** При хорошей настройке заявлено 80–90%; улучшается с конкретными промптами и примерами в `.cursorrules`.
- **Реальные данные (API, БД)?** Да, через скрипты/интеграции, которые создаёт AI (например, Google Sheets → отчёт в markdown).
- **Безопасность?** Доступ через GitHub (org/repo), аудит, клиентские данные — в отдельных репо; в `.cursorrules` можно описать политики данных.
- **Масштаб?** Малая компания — 2–3 отдела; крупная — больше репо и более жёсткий контроль доступа.

## 10. Связь с вашим воркспейсом (Cursor)

У вас уже есть:

- **Work/Задачи/** — планы, задачи, ADHD-friendly процесс (rules).
- **Projects/** — FinanceBot, VoiceBot, CreatorBot и др.
- **.cursor/rules/** — стандарты кода, задачи, Finance Bot, workflow.

Идеи из шаблона, которые можно точечно перенести:

1. **Отдельные .cursorrules по «контекстам»** — не обязательно по отделам; например, «контекст задач», «контекст FinanceBot», «контекст документации» с явным переключением через фразы.
2. **Один канонический источник и [REF:]** — для важных решений и спецификаций в Documentation/ и проектах.
3. **Скрипты clone/update** — если появятся отдельные репо под проекты или под «отделы» (например, Work, Projects, Scripts как отдельные репо) — те же паттерны setup.
4. **Правила для агента по git** — не пушить без подтверждения, предлагать коммиты — уже частично пересекаются с вашими workflow rules.

Если нужно, могу предложить конкретный план: какие папки считать «контекстами», где добавить `.cursorrules` и какие фразы использовать для переключения в вашей текущей структуре.
